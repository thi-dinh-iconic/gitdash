# 4.2 Anomaly Detection

## Summary
Added lightweight statistical anomaly detection to the workflow detail page. Uses a rolling baseline (mean + stddev from preceding runs) to flag runs whose duration or queue wait is significantly outside the normal range (> 2 standard deviations). All computation is client-side — no new API endpoints.

## Features
- **Anomaly badges on Runs table:** Each run row in the Runs tab shows an "Anomaly" badge if the run's duration or queue wait is a statistical outlier. Badge colour reflects severity (moderate/high/extreme). Hovering shows a tooltip with the exact z-score and comparison to baseline.
- **Anomaly summary in Reliability tab:** A dedicated section lists all detected anomalies (up to 15), sorted by severity. Each entry shows the run number, metric details, and severity level.
- **Causal detection:** Each run's baseline is computed only from *preceding* runs (not including itself), so detection is causal — it only uses data available at the time the run completed.
- **Three severity levels:**
  - Moderate: 2-3 stddev from mean (amber)
  - High: 3-4 stddev from mean (orange)
  - Extreme: 4+ stddev from mean (red)

## Algorithm
- Rolling window of 20 preceding runs for baseline computation
- Minimum 5 samples required before detection activates
- Z-score = (value - mean) / stddev
- Duration: flagged if z > 2 (slow) or z < -2 (unusually fast)
- Queue wait: only flagged if z > 2 (long wait; short wait is fine)
- Guard against false positives: skips when stddev < 1ms (all values identical)
